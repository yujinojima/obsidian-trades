# CLAUDE.md — Trading Research Wiki Schema

This file is the instruction manual for any Claude session that reads, writes, or maintains this vault.
It is **not** a trade report. It is the rulebook. Read it first before touching anything.

---

## 1. What this vault is

A **research wiki** for a crypto backtesting and hypothesis-learning system. The underlying engine
(freqtrade + the `research/` pipeline) produces trade-level data. This vault turns that data into
durable, linkable knowledge: strategies, trades, concepts, primitives, market states, exit diagnoses,
and open questions — maintained by Claude, readable by Obsidian.

Reference architecture: Karpathy-style LLM wiki.
- `raw/` = immutable source of truth (never edit after placement).
- `wiki/` = generated, Claude-maintained markdown.
- `index.md` = the content-oriented map across `wiki/`.
- `log.md` = append-only chronological record of every ingest / maintenance pass.
- `dashboard.md` = operational view (latest strategy, open questions, lint status).

---

## 2. Hard rules

1. **Never edit files under `raw/` after they land.** They are the provenance. If you need to
   correct a raw export, re-run the upstream export and add a new dated folder under
   `raw/journals/`; do not mutate an existing one.
2. **Every new ingest must update `index.md` and append to `log.md`.** No silent additions.
3. **Every strategy gets exactly one page** under `wiki/strategies/<StrategyName>.md`.
4. **Every trade gets exactly one page** under `wiki/trades/<StrategyName>/trade_NNNN.md`, with
   YAML frontmatter and a chart embed from `raw/charts/`.
5. **Recurring ideas become concept pages** under `wiki/concepts/<slug>.md` (e.g. `missed-continuation`,
   `mfe-capture-ratio`). If you mention a concept more than twice across trade/strategy pages,
   it needs a concept page.
6. **Primitives use canonical registry names** (hyphenated, matching
   `freqtrade/user_data/prim_registry.json`). Use frontmatter `aliases:` for human-friendly names
   (e.g. `liquidity-sweep-reversal.md` has aliases `[liquidity-sweep, sweep]`).
7. **Contradictions and weak evidence are flagged, not hidden.** Use a `> [!warning]` callout.
   Sample size < 20 → mark confidence `LOW`. Sample size < 5 → add `> [!caution] Provisional`.
8. **No lookahead bias** in any "setup explanation" or "signal breakdown" section. Only
   information available *strictly before* `open_date` may appear. The trade export has already
   enforced this for exported trades — do not re-derive signals from post-entry data.
9. **Do not hallucinate signal causes.** If a primitive firing is unavailable (because signal
   export is not wired), write `Unavailable — awaiting signal export`. Do not infer it from price.
10. **Mark unavailable data.** A primitive page with no observations should say
    `Status: provisional — no registered firings yet.` A market-state page with no labelled
    trades should say `Status: unlabelled — awaiting Phase 1 market-state labeller`.

## 3. File layout

```
obsidian_trade_journals/
├── CLAUDE.md                  ← this file
├── index.md                   ← global content map
├── log.md                     ← append-only activity log
├── dashboard.md               ← operational dashboard
├── raw/                       ← IMMUTABLE
│   ├── backtests/             ← raw freqtrade backtest JSON/zip
│   ├── journals/              ← v1 exported strategy folders (dated)
│   ├── signals/               ← (future) prim firings export
│   └── charts/                ← PNG charts referenced by trade pages
├── wiki/                      ← Claude-maintained
│   ├── strategies/            ← one page per strategy
│   ├── trades/<Strat>/        ← one page per trade
│   ├── concepts/              ← recurring ideas (missed-continuation, etc.)
│   ├── primitives/            ← registry-canonical names + aliases
│   ├── market-states/         ← regime pages (trending-up, compression, …)
│   ├── exit-analysis/         ← per exit-reason and per exit-diagnosis
│   ├── questions/             ← open research questions
│   └── synthesis/             ← current trading thesis, rolling
├── reports/
│   └── wiki_lint_report.md    ← generated by scripts/wiki_maintain.py
└── scripts/
    └── wiki_maintain.py       ← rebuild index, lint, append to log
```

## 4. Maintenance pass — required after any ingest

Run `python3 scripts/wiki_maintain.py` after every new strategy export. It will:

1. Rebuild `index.md` from a filesystem scan of `wiki/`.
2. Append a timestamped entry to `log.md`.
3. Emit `reports/wiki_lint_report.md` covering:
   - orphan pages (no inbound links)
   - broken `[[wikilinks]]`
   - pages missing YAML frontmatter
   - stale pages (`updated` > 30 days old)
   - concept mentions without a concept page
   - strategies without trade links
   - trades without related-trade links

## 5. When to create new concept / primitive / market-state pages

| Trigger | Action |
|---|---|
| An outcome label (`missed_continuation`, `slow_loss`, …) appears in ≥ 2 trade pages and has no concept page | Create `wiki/concepts/<outcome-label>.md` |
| An exit diagnosis (`premature_exit`, `poor_entry`, …) appears in ≥ 2 trade pages | Create `wiki/exit-analysis/<diagnosis>.md` |
| A primitive is mentioned in a signal breakdown | Ensure `wiki/primitives/<registry-name>.md` exists (stub if needed) |
| A market state (`trending_up`, `compression`, …) appears in entry metadata | Ensure `wiki/market-states/<state>.md` exists |
| A question arises that research data cannot yet answer | Create `wiki/questions/<slug>.md` |

---

## 6. YAML frontmatter contract

Every `wiki/**/*.md` page must start with frontmatter. Minimum fields:

```yaml
---
title: <Human-readable title>
type: strategy | trade | concept | primitive | market-state | exit-diagnosis | question | synthesis
updated: <YYYY-MM-DD>
status: active | provisional | stub | deprecated
---
```

Page-type extensions:
- `trade`: add `strategy`, `pair`, `open_date`, `close_date`, `profit_ratio`, `outcome`, `exit_diagnosis`
- `primitive`: add `registry_name`, `aliases`, `tier`, `timeframes`
- `strategy`: add `total_trades`, `win_rate`, `confidence`
- `concept`: add `evidence_count` (number of trade pages citing it)

---

## 7. Non-goals

- No database, no vector search, no web app.
- No live trading layer.
- No LLM-API-based generation from this vault (vault is a consumer, not a driver).
- No sync to the user's live Obsidian vault (`~/Documents/Obsidian Vault/`) until
  `scripts/sync_to_vault.py` is added and explicitly run.

---

## 8. Integration with the research pipeline

See the sibling `research/` project (planned, not yet built):

- Phase 1 `research/src/labels/market_state.py` → populates real status for `wiki/market-states/*.md`
- Phase 2 `research/src/labels/outcome_labeller.py` → emits `outcome_R`, `MFE_R`, `MAE_R` that land in
  trade-page frontmatter
- Phase 3 `research/src/discovery/abductive.py` → produces `discovered_patterns.csv` which feeds
  `wiki/synthesis/current-trading-thesis.md` and may spawn new concept pages
- Phase 4 walk-forward → promotes cells to `research_candidate` tier in the prim registry; those
  promotions add a frontmatter `tier:` bump to the corresponding `wiki/primitives/*.md`

Until those ship, primitive and market-state pages remain **provisional**.

---

## 9. Commit discipline

- One commit per ingest + maintenance pass.
- Commit message: `wiki: ingest <StrategyName> <YYYY-MM-DD>` or `wiki: maintain <YYYY-MM-DD>`.
- Never force-push. Never rewrite history. The vault is append-friendly by design.
